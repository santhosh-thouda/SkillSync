10. AUTH SERVICE TESTING
------------------------
Base URL direct:
- http://localhost:8081

Gateway URL:
- http://localhost:8080/auth

Endpoints:
- POST /auth/register
- POST /auth/login
- POST /auth/refresh

Register request:
POST http://localhost:8080/auth/register
Content-Type: application/json

{
  "name": "Test User",
  "email": "testuser@example.com",
  "password": "Password123",
  "role": "LEARNER"
}

Expected:
- 201 Created
- "User registered successfully"

Login request:
POST http://localhost:8080/auth/login

{
  "email": "testuser@example.com",
  "password": "Password123"
}

Expected:
- 200 OK
- token returned
- userId returned
- role returned

Refresh request:
POST http://localhost:8080/auth/refresh

{
  "token": "PASTE_LOGIN_TOKEN"
}

Expected:
- 200 OK
- new token returned

JWT claim expectations:
- token contains subject = user email
- token contains userId claim
- token contains role claim
- these claims are used by downstream services for authorization and ownership checks

Validation tests:
- blank email
- invalid email
- blank password
- missing role

Expected:
- 400 Bad Request
- response body contains validationErrors

Security filter test:
1. Call protected endpoint without Authorization header
2. Expect 401 or access denied
3. Call same protected endpoint with:
   Authorization: Bearer <jwt-token>
4. Expect request to pass


10A. ROLE / ACCESS CONTROL MATRIX
---------------------------------
Use at least these three users for manual testing:
- ADMIN user
- LEARNER user
- MENTOR user

Intended access rules currently implemented:
- ADMIN:
  full access to protected user, mentor, skill, session, group, and review operations
- LEARNER:
  can read/update/delete own user profile
  can create own group and join/leave only as self
  can create review only with own userId
  can request session only with own learnerId
  can cancel own session
  can read own learner sessions
- MENTOR:
  can apply for mentor profile only with own userId
  can update/delete own mentor profile
  can accept/reject/complete own mentor sessions
  can read own mentor sessions
  can create skills

Expected security status codes:
- 401 when Authorization header is missing or token is invalid
- 403 when token is valid but role or ownership is wrong

How queue, exchange, and routing key are configured in this project:
1. Open:
   - C:\Users\santh\Advance_Java_MicroServices\SkillSync\session-service\src\main\resources\application.properties
2. Check these properties:
   - rabbitmq.exchange
   - rabbitmq.queue
   - rabbitmq.routingkey
3. These values are used by session-service when publishing messages
4. notification-service should listen to the same queue/binding setup

Meaning of each RabbitMQ property:
1. Exchange
   - receives messages from the producer
2. Routing key
   - decides how the message is routed
3. Queue
   - stores the routed message until a consumer reads it

How to add queue, exchange, and routing key in configuration:
1. In application.properties of the producer/consumer service add:
   - rabbitmq.exchange=session.exchange
   - rabbitmq.queue=session.queue
   - rabbitmq.routingkey=session.routingkey
2. In Java RabbitMQ config class, bind them together
3. Producer publishes to:
   - exchange + routing key
4. Consumer listens on:
   - queue

Example RabbitMQ property configuration:
   rabbitmq.exchange=session.exchange
   rabbitmq.queue=session.queue
   rabbitmq.routingkey=session.routingkey

How they work together:
1. session-service sends message to exchange:
   - session.exchange
2. using routing key:
   - session.routingkey
3. RabbitMQ routes the message to queue:
   - session.queue
4. notification-service consumes from:
   - session.queue
